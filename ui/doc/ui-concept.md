# UI Concept: Protocol-Driven ERP

**Дата**: 2026-04-03
**Статус**: Концепция, крупные мазки

---

## 1. Принцип

UI — это **проекция** трёх серверных контрактов. Клиент (web, mobile, desktop) — **рендерер**, не приложение. Бизнес-поведение управляется из одного места (Rust), клиент только отображает уже разрешённое состояние.

---

## 2. Три контракта

```
                         Rust (axum)
            ┌───────────────────────────────┐
            │          ui_runtime           │
            │  (resolve metadata + state    │
            │   + permissions → contracts)  │
            └──────┬────────┬────────┬──────┘
                   │        │        │
              ┌────▼──┐ ┌───▼───┐ ┌──▼──────┐
              │Screen │ │ Data  │ │Behavior │
              │Descr. │ │Payload│ │ State   │
              └───┬───┘ └───┬───┘ └────┬────┘
                  │         │          │
                  └─────────┼──────────┘
                            │ JSON
                     ┌──────▼──────┐
                     │   Client    │
                     │  (renderer) │
                     └─────────────┘
```

### 2.1. ScreenDescriptor — что показать

Структура экрана: секции, поля, колонки, действия, layout. Загружается один раз при открытии экрана. Не содержит данных и не содержит состояния доступности.

### 2.2. DataPayload — данные

Бизнес-данные: запись, строки грида, справочные значения. Приходят отдельно от экрана. Перезагружаются при навигации, сохранении, фильтрации.

### 2.3. BehaviorState — что можно

Разрешённое состояние: какие поля editable/readonly/hidden, какие действия enabled/disabled, какие валидации активны, какие формулы вычислять. **Меняется** при изменении данных, статуса, роли.

- **Ribbon** — динамический. Проецирует `actions[]` из ScreenDescriptor активной вкладки. Enabled/disabled из BehaviorState.

## 4. Примитивы

Клиент умеет рендерить ровно 6 типов секций. Любой ERP-экран — композиция из них.

| Примитив | Что рендерит | Примеры экранов |
|---|---|---|
| **Collection** | Таблица/грид/дерево с данными. Inline editing, виртуализация, keyboard nav | Список документов, строки документа, остатки, матрица прав |
| **Record** | Форма с полями по секциям | Карточка документа, карточка контрагента, настройки |
| **Selector** | Выбор связанной сущности. Combo + popup | Выбор склада, номенклатуры, сотрудника |
| **ActionBar** | Группа действий. Кнопки, split-buttons, meню | Toolbar формы, действия строки, workflow-кнопки |
| **Feedback** | Ответ системы | Валидация, toast, confirm, progress, пустое состояние |
| **Workspace** | Shell: навигация, вкладки, контекст | Sidebar, tab bar, breadcrumbs, user menu |

Master-detail = `Collection + Record`. Tree-grid = `Collection` в режиме hierarchy. Многовкладочный документ = `Record` с несколькими секциями. Wizard = `Record + Feedback` с шагами.

## 5. Behavior Protocol

Клиент **не решает**, что можно и что нельзя. Он спрашивает сервер.

```
   Client                                  Server (ui_runtime)
     │                                          │
     │  1. open_screen("move_material", id=42)  │
     │ ────────────────────────────────────────► │
     │                                          │
     │  2. ScreenDescriptor + DataPayload       │
     │     + BehaviorState                      │
     │ ◄──────────────────────────────────────── │
     │                                          │
     │  (user меняет поле status)               │
     │                                          │
     │  3. event { field_changed: "status" }    │
     │ ────────────────────────────────────────► │
     │                                          │
     │  4. BehaviorState (пересчитанный)        │
     │     + UiEffect[] (toast, refresh...)     │
     │ ◄──────────────────────────────────────── │
     │                                          │
     │  (user жмёт "Провести")                  │
     │                                          │
     │  5. action { id: "execute" }             │
     │ ────────────────────────────────────────► │
     │                                          │
     │  6. DataPayload (обновлённые данные)     │
     │     + BehaviorState (executed: readonly)  │
     │     + UiEffect[toast("Проведено")]       │
     │ ◄──────────────────────────────────────── │
```

Один и тот же протокол для web, mobile, desktop. Поведение формы гарантированно одинаковое на всех платформах, потому что оно вычисляется в одном месте — `ui_runtime`.

## 6. Граница server / client

| Отвечает сервер (ui_runtime) | Отвечает клиент (рендерер) |
|---|---|
| Доступность действий (enabled/disabled/hidden) | Focus management (Tab, Enter, Escape) |
| Видимость полей и секций | Keyboard shortcuts (Ctrl+S → отправить action) |
| Read-only / editable по статусу и роли | Virtual scroll / lazy rendering |
| Workflow transitions | Drag-drop interaction |
| Валидация бизнес-правил | Undo/redo в рамках сессии |
| Side effects (UiEffect[]) | Анимации, transitions |
| Формулы вычисляемых полей (декларативно) | Исполнение формул локально (instant preview) |

**Правило**: если решение зависит от роли, статуса или бизнес-правила — сервер. Если от позиции курсора или UX-feedback — клиент.

## 6.1. Закон: Universal Behavior Cycle

> **BehaviorState — единый паттерн управления состоянием для ВСЕХ объектов в системе.**

Каждый экран — документ, справочник, отчёт, список — живёт по одному циклу:

```
load → render(ScreenDescriptor + Data + BehaviorState)
         │
         ├── user action (click, edit, save, execute, delete...)
         │         │
         │         ▼
         │   server: handleAction(screenId, actionId, data)
         │         │
         │         ▼
         │   ActionResult { data?, behavior?, effects[] }
         │         │
         │         ▼
         └── apply effects (toast, close, navigate)
             apply new data + behavior
             re-render
```

**Нет исключений.** Не бывает "этот экран управляет своим состоянием по-другому":
- Документ (draft → posted → cancelled) — BehaviorState переключает поля и действия
- Справочник (active / inactive) — BehaviorState делает карточку readonly
- Список — BehaviorState определяет доступные кнопки
- Дерево — BehaviorState определяет можно ли перемещать узлы
- Отчёт — BehaviorState может быть пустым (всё readonly, один action "Обновить")

**Клиент не содержит if/else по типу объекта.** Он рендерит то, что пришло.

### Web-клиент (стартовый)

| Слой | Технология | Зачем |
|---|---|---|
| Язык | TypeScript | Типизация протокола |
| Сборка | Vite | Быстро, просто |
| Grid | AG Grid | Collection с inline editing, виртуализация, keyboard |
| CSS | Tailwind CSS | Утилитарный, дизайн-токены |
| Рендеринг | Vanilla TS / Web Components | Метаданные → DOM напрямую, без промежуточного фреймворка |
| Push | WebSocket | Уведомления, real-time BehaviorState |

Формы рендерятся **динамически из ScreenDescriptor** — не нужен React/Vue для генерации одинаковых полей. Рендерер — маленький движок, который читает метаданные и создаёт DOM.

AG Grid — единственная тяжёлая зависимость. Остальное — свой код, полный контроль.
